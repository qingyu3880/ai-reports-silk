## 一、日报详细要求

5. **核心洞察** (3条)
   - 深度分析
   - 趋势判断
   - 影响评估

### 4. 格式要求
- Markdown格式
- 使用emoji图标
- 表格清晰
- 链接可点击
- 字数: 2000-3000字

### 5. 文件保存
- **路径**: `/root/.openclaw/workspace/reports/ai-daily-YYYY-MM-DD.md`
- **命名**: `ai-daily-YYYY-MM-DD.md`

## 三、日报执行流程

### 阶段1: 生成前检查
```bash
# 1. 运行前置检查
bash scripts/mandatory-pre-check.sh

# 2. 选择任务类型: 1 (内容生成)

# 3. 确认理解要求
```

### 阶段2: 生成内容
- 使用提示词生成日报
- 保存到正确路径
- 验证文件存在且大小>1KB

### 阶段3: 用户交付（关键）
```bash
# 1. 统计行数
wc -l < reports/ai-daily-YYYY-MM-DD.md

# 2. 完整推送到对话框
# 如果>100行，分段推送

# 3. 询问用户确认
echo "完整日报已推送，请确认："
echo "1. 是否收到？"
echo "2. 内容是否完整？"
echo "3. 格式是否正确？"
```

### 阶段4: 持久化
```bash
# 1. 运行sync脚本
python3 sync_reports.py

# 2. Git提交
git add reports/ai-daily-YYYY-MM-DD.md dashboard/reports.json
git commit -m "新增: AI日报 YYYY-MM-DD"

# 3. Git推送
git push origin master:main

# 4. 验证推送成功
git log origin/main..master --oneline  # 应为空
```

### 阶段5: 完成确认
- 用户明确表示"收到"或开始讨论
- 任务才算真正完成

## 四、所有教训总结

### Error #001: write工具缺少path参数
**原因**: 记错参数名
**修复**: 始终使用`path`参数

### Error #002: 假设GitHub仓库存在
**原因**: 没有验证前置条件
**修复**: 先验证再操作

### Error #003: git push超时
**原因**: 网络问题，没有重试
**修复**: 设置超时，自动重试

### Error #004: 定时任务虚假完成报告
**原因**: 验证逻辑错误，依赖日志
**修复**: 验证文件系统真实状态

### Error #005: 报告未推送到GitHub
**原因**: 推送超时，缺少验证
**修复**: 推送后验证远程状态

### Error #006: 日报未推送到对话框
**原因**: 流程不完整，只提交到GitHub
**修复**: 用户必须在对话中看到内容

### Error #007: 日报未推送到用户对话框
**原因**: 误解完成标准
**修复**: 对话是主要交付渠道

### Error #008: 只推送概要，未推送完整报告
**原因**: 没有验证推送内容的完整性
**修复**: 推送前统计行数，推送后对比

## 五、修复方案汇总

### 1. 强制前置检查
**文件**: `scripts/mandatory-pre-check.sh`
**作用**: 每次任务前必须运行
**内容**:
- 读取标准流程
- 检查今日错误
- 确认理解核心原则
- 任务类型特定要求

### 2. 简化执行清单
**文件**: `SIMPLE_CHECKLIST.md`
**原则**:
- 不打勾不能继续
- 用户确认才算完成
- 出错立即停止

### 3. 自动验证脚本
**文件**: `scripts/auto-verify-push.sh`
**作用**: 推送后自动检查
- 统计行数
- 对比源文件
- 长内容警告

### 4. 防错机制
**文件**: `skills/anti-error-mechanism/SKILL.md`
**核心**:
- Skill是强制规则，不是参考
- 自动验证，不依赖人工
- 用户确认才算完成

### 5. 每日自检
**文件**: `scripts/daily-self-check.sh`
**作用**: 每天早上自动检查系统状态

## 六、核心原则（必须遵守）

1. **不靠记忆，靠文件**
   - 所有信息写入文件
   - 使用memory_search查找

2. **不假设，只验证**
   - 验证前置条件
   - 验证执行结果
   - 验证用户收到

3. **完成意味着用户收到**
   - 不是"我做了"
   - 是"用户收到了"
   - 必须用户确认

4. **强制执行力**
   - 前置检查必须运行
   - 清单不打勾不能继续
   - 不确认不算完成
