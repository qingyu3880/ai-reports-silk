# Skill: 用户交付标准

## 核心原则

**用户必须在对话中看到结果**

GitHub和网站是备份，对话是主要交付渠道。

---

## 交付渠道优先级

| 优先级 | 渠道 | 用途 | 必须? |
|-------|------|------|-------|
| 1 | **对话框** | 用户直接阅读 | ✅ 必须 |
| 2 | **GitHub** | 备份/版本控制 | ✅ 必须 |
| 3 | **Vercel网站** | 公开访问/分享 | ⚪ 可选 |

---

## 内容生成任务完成标准

### 必须完成的步骤

#### 1. 生成内容 ✅
- 文件保存到正确路径
- 文件大小合理（>1KB）
- 内容格式正确

#### 2. 推送到对话框 ✅ (关键)
- **完整内容**显示在对话中
- 格式正确（Markdown渲染）
- 用户可以直接阅读

#### 3. 用户确认 ✅ (关键)
- 用户表示"收到了"
- 或用户开始阅读/讨论内容
- 或用户没有提出异议

#### 4. Git提交 ✅
- git add
- git commit

#### 5. Git推送 ✅
- git push origin master:main
- 验证推送成功

#### 6. 同步到网站 ⚪ (可选)
- 运行 sync_reports.py
- 验证网站更新（如配置）

## 检查清单（每个内容任务）

```markdown
## 内容交付检查清单

### 生成阶段
- [ ] 文件已生成
- [ ] 内容格式正确
- [ ] 文件大小 > 1KB

### 用户交付阶段（关键）
- [ ] 内容已推送到对话框
- [ ] 格式渲染正确
- [ ] 用户能看到完整内容
- [ ] 用户确认收到（或没有异议）

### 持久化阶段
- [ ] Git已提交
- [ ] Git已推送
- [ ] 验证远程状态

### 可选阶段
- [ ] 同步到网站（如需要）
```

## 常见错误

### 错误1: 只提交到GitHub，不推送到对话框
**症状**: 用户说"我没有收到"
**修复**: 立即补发到对话框
**预防**: 严格按照检查清单执行

### 错误2: 推送了链接而不是内容
**症状**: 用户需要点击链接才能看
**修复**: 直接推送完整内容
**预防**: 对话中显示完整内容，链接作为补充

### 错误3: 推送了摘要而不是完整内容
**症状**: 用户说"内容不全"
**修复**: 补发完整内容
**预防**: 确保推送的是完整内容

## 补救流程

如果用户说"没有收到":

1. **立即道歉**: "抱歉，我立即补发"
2. **完整推送**: 将内容完整推送到对话框
3. **确认收到**: "请确认是否收到完整内容"
4. **记录错误**: 更新ERROR_LOG.md
5. **优化流程**: 防止再次发生
